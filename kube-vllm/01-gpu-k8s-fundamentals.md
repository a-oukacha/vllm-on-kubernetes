# 01 - GPU Kuberentes Fundamentals

> **Scope:** how Kubernetes sees, schedules, and isolates GPUs - and the four ways to share one.
> Everything here is the substrate the rest of the series stands on.

## Why GPU nodes are different

Kubernetes natively understands two compressible/incompressible resources: CPU and memory.
A GPU is neither - it's an opaque PCIe device exposed to containers through three cooperating layers:

```
NVIDIA kernel driver        -> makes the GPU usable on the host
nvidia-container-toolkit    -> injects the GPU + libraries into a container (via CDI today)
device plugin / DRA driver  -> tells the kubelet "this node has N GPUs", allocates them to pods
```

If any layer is missing you get the classic symptoms: `nvidia.com/gpu` absent from node
capacity, or a pod that schedules but whose `nvidia-smi` returns "no devices found".

**Senior DevOps tip:** 90% of "GPU not working" tickets are one of three things - driver
not loaded (`nvidia-smi` fails *on the host*), toolkit not wired into containerd (`nvidia`
runtime missing from `/etc/containerd/config.toml`), or the device-plugin DaemonSet not
Running on that node. Check them in that order; don't touch pod YAML until the node is healthy.

---

## CDI - how GPUs actually get into containers now

Modern stacks use the **Container Device Interface (CDI)** rather than the legacy
`nvidia` OCI hook. CDI is a vendor-neutral spec: a JSON/YAML file describes which device
nodes, libraries, and env vars to inject. containerd ≥ 1.7 and CRI-O support it natively.

```bash
# CDI spec generated by the toolkit (GPU Operator does this for you)
ls /etc/cdi/ /var/run/cdi/
# nvidia.yaml   <- lists nvidia.com/gpu=all, gpu=0, gpu=1 ...

# Verify containerd has CDI enabled
grep -A2 'enable_cdi' /etc/containerd/config.toml
```

**Senior Dev tip:** CDI is why you no longer set `runtimeClassName: nvidia` in most
clusters - requesting `nvidia.com/gpu` is enough. If you *do* still see `runtimeClassName`
in older manifests, it's a legacy path; the GPU Operator's default is now CDI injection.

## Requesting a whole GPU

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: vllm-smoke
spec:
  restartPolicy: Never
  containers:
    - name: vllm
      image: vllm/vllm-openai:v0.9.2      # pin - never :latest in prod
      command: ["nvidia-smi"]
      resources:
        limits:
          nvidia.com/gpu: 1               # whole, exclusive GPU
          memory: "24Gi"
          cpu: "8"
        requests:
          nvidia.com/gpu: 1
          memory: "24Gi"
          cpu: "8"
```

**Hard rules of `nvidia.com/gpu`:**
- It is an **extended resource**: limit must equal request, and it cannot be over-committed.
- A whole GPU is **exclusive** to one container - no native fractional sharing (see below).
- It is **not** topology-aware by itself; two requested GPUs may land on opposite sides of the
 PCIe/NVLink fabric unless you help the scheduler (Operator GPU Feature Discovery + affinity).

**Senior Dev tip:** set `requests == limits` for CPU and memory too on GPU pods. A vLLM
pod that gets CPU-throttled during the tokenizer/prefill path will show up as mysterious TTFT
spikes that look like a GPU problem but aren't. Give it real CPU (8-32 cores) and pinned memory.

## The four ways to share a GPU

A single H100 is wasted on a 1B reranker. These are your sharing options, weakest to strongest isolation:

| Mechanism | Isolation | Memory protection | Best for | Exposed as |
|---|---|---|---|---|
| **Time-slicing** | None (round-robin context switch) | no shared, can OOM each other | Dev, bursty low-QPS, demos | `nvidia.com/gpu` (oversubscribed) |
| **MPS** (Multi-Process Service) | Spatial, soft | optional memory caps | Many small co-operative models | `nvidia.com/gpu` (MPS-managed) |
| **MIG** (Multi-Instance GPU) | Hardware partition | yes hard, isolated | Multi-tenant, predictable SLO | `nvidia.com/mig-1g.10gb` etc. |
| **DRA** (Dynamic Resource Allocation) | Depends on backing mode | Depends | The *future* default; flexible claims | `resourceClaim` (not a count) |

### Time-slicing (cheapest, no isolation)
Configured in the GPU Operator's device-plugin config - one physical GPU advertised as N "replicas":
```yaml
# device-plugin config (ConfigMap consumed by the Operator)
sharing:
  timeSlicing:
    resources:
      - name: nvidia.com/gpu
        replicas: 4        # 1 GPU now schedules 4 pods
```
> All four pods share memory with **zero protection**. One model that grows its KV cache can
> OOM its neighbors. Never use for production multi-tenant.

### MIG (hard isolation, Ampere/Hopper/Blackwell)
Partitions one A100/H100 into up to 7 isolated instances with dedicated SMs + memory + cache:
```bash
# Common A100 80GB / H100 profiles
# 1g.10gb -> 7 instances   2g.20gb -> 3   3g.40gb -> 2   7g.80gb -> full GPU
nvidia-smi mig -lgip
```
Pods then request the *profile* as a resource:
```yaml
resources:
  limits:
    nvidia.com/mig-1g.10gb: 1
```

### DRA - where Kubernetes GPU scheduling is heading
**Dynamic Resource Allocation** (beta in k8s 1.32) replaces "count of opaque devices" with
*structured claims* - you ask for "a GPU with ≥ 40GB and MIG-capable", and a **DRA driver**
(NVIDIA ships one) matches it. This finally makes fractional/MIG/topology requests first-class.

```yaml
apiVersion: resource.k8s.io/v1beta1
kind: ResourceClaimTemplate
metadata:
  name: single-gpu
spec:
  spec:
    devices:
      requests:
        - name: gpu
          deviceClassName: gpu.nvidia.com
          selectors:
            - cela:               # CEL expression
                expression: "device.capacity['memory'].compareTo(quantity('40Gi')) >= 0"
---
# pod references the claim instead of a resource count
spec:
  resourceClaims:
    - name: gpu
      resourceClaimTemplateName: single-gpu
  containers:
    - name: vllm
      resources:
        claims:
          - name: gpu
```

**Architect tip:** don't migrate production to DRA yet (it's beta, and the GPU Operator's
device-plugin path is still the battle-tested default), but **design your platform abstraction
so the request mechanism is swappable.** Wrap "I need a GPU of class X" in a Helm value, not a
hardcoded `nvidia.com/gpu: 1`. When DRA goes GA you change one template, not 200 manifests.

**Senior DevOps tip:** pick *one* sharing strategy per node pool, never mix. A node that
is simultaneously time-sliced and MIG-enabled produces scheduling decisions nobody can reason
about. Label node pools by strategy (`gpu-sharing=mig`, `gpu-sharing=exclusive`) and route with affinity.

## Keeping the wrong workloads off expensive nodes

GPU nodes cost 10-40× a CPU node. Two complementary fences:

**1. Taint the node** so only GPU-tolerating pods land:
```bash
kubectl taint node gpu-node-1 nvidia.com/gpu=present:NoSchedule
```
```yaml
tolerations:
  - key: nvidia.com/gpu
    operator: Exists
    effect: NoSchedule
```

**2. Target the right GPU SKU** with affinity (labels come free from GPU Feature Discovery - doc 02):
```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
        - matchExpressions:
            - key: nvidia.com/gpu.product
              operator: In
              values: ["NVIDIA-H100-80GB-HBM3", "NVIDIA-A100-SXM4-80GB"]
```

**Senior DevOps tip:** taints stop *other* workloads landing on GPU nodes; they do **not**
stop a GPU pod from landing on the *wrong* GPU node. You need both the toleration (to get on
GPU nodes at all) and affinity (to get the right SKU/memory). Forgetting the second is why a
70B job lands on a 40GB card and CrashLoops with CUDA OOM.

## Topology matters before you even start (multi-GPU)

Two GPUs in one pod is only fast if they share NVLink. Across the wrong PCIe switch, tensor
parallelism collapses. Check the fabric:
```bash
kubectl exec -it <pod> -- nvidia-smi topo -m
#        GPU0 GPU1 GPU2 GPU3
# GPU0    X   NV12 NV12 NV12     <- NV = NVLink (good for TP)
# GPU1   NV12  X   NV12 NV12
# ...
# Legend: NV# = NVLink, PIX/PXB/SYS = PCIe (progressively worse)
```

**Architect tip: the unit of capacity planning for large models is not the GPU, it's
the NVLink island.** An 8×H100 SXM node is one island (fast TP up to 8). Two 4-GPU PCIe boxes
are *not* equivalent to one 8-GPU SXM box even with the same GPU count - see doc 06. Buy/rent
topology, not just FLOPs.

## Inspecting GPU state

```bash
# Capacity & allocatable per node
kubectl describe node gpu-node-1 | grep -A6 'Allocatable'

# Who holds GPUs right now
kubectl get pods -A -o json | jq -r '
  .items[]
  | select(.spec.containers[].resources.limits["nvidia.com/gpu"])
  | "\(.metadata.namespace)/\(.metadata.name)"'

# Live utilization inside a pod
kubectl exec -it <pod> -- nvidia-smi --query-gpu=index,name,memory.used,memory.total,utilization.gpu --format=csv
```

## Common issues at this layer

| Symptom | Root cause | First check |
|---|---|---|
| Pod `Pending`, `Insufficient nvidia.com/gpu` | No free GPU or missing toleration | `kubectl describe node`; taints vs tolerations |
| `nvidia.com/gpu` absent from node capacity | Device plugin / DRA driver not running | DaemonSet in `gpu-operator` ns |
| Container starts but `nvidia-smi`: no devices | Toolkit/CDI not injecting | `/etc/cdi/nvidia.yaml`, containerd CDI enabled |
| Schedules but CUDA error / version mismatch | Image CUDA newer than node driver | Match vLLM image CUDA <-> driver branch |
| 2 GPUs allocated but TP is slow | GPUs on different PCIe switches | `nvidia-smi topo -m` - want NVLink |
| Neighbors randomly OOM | Time-slicing with no memory protection | Move to MIG, or stop oversubscribing |
